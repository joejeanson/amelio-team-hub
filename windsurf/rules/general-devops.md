## 1. Commit Strategy
- **Language**: **English ONLY**.
- **Format**: `<type>(#TASK): <Description>`
- **Example**: `feat(#4321): Implement user login screen`
- **Types**:
  - `feat`: New feature
  - `fix`: Bug fix
  - `refactor`: Restructuring code (no behavior change)
  - `chore`: Build, deps, tool config
  - `docs`: Documentation updates
  - `test`: Adding/Fixing tests
  - `style`: Formatting, semi-colons
  - `perf`: Performance improvements
- **Content**: Max 3 bullet points summarizing changes. Reference User Story IDs (e.g., "US: #1234").

## 2. Pull Requests (PR)
- **Description**:
  - Follow the template.
  - Bullet points for changes.
  - **No Blame**: Never name developers or assign blame in descriptions.
- **Verification**:
  - **100% Test Coverage** is mandatory before opening.
  - Self-review against `frontend-react.md` and `backend-dotnet.md`.

## 3. Work Items (Azure DevOps)
- **Bug Fields**:
  - Cause of bug: `Custom.Causeofbug`
  - Resolution: `Microsoft.VSTS.Common.Resolution`
  - Origin Developer: `Custom.OriginDeveloper`
  - Found In Environments: `Custom.FoundInEnvironments`
- **Status**: Do not manually change status (handled by PR closure usually).

## 4. Documentation & Generated Files
- All technical documentation, analysis, and architectural decisions must be written in **English**.
- ALL files generated by Cascade (reports, coverage comparisons, markdown files, code comments, etc.) must be written in **English** ‚Äî the "respond in French" rule applies ONLY to chat messages, never to files written to disk.
